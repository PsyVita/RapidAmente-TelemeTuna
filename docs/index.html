RAPIDAMENTE · EV RACE TELEMETRY

TelemeTuna v1.1

Feed the tuna every frame, clean telemetry wins the game. TelemeTuna catches the radio stream off the Rapidamente electric race car, repairs what the radio mangled, logs everything it touched, and turns it into live dashboards the whole team can open from anywhere.

🐠 I just want to watchNo install. Get the links, log in, read the dashboard properly. 🐙 I'm setting it upInstall, deploy to the cloud, feed data in, keep it healthy. 🐋 Show me how it worksEvery service, every pipeline station, every design decision.

1. Overview

What TelemeTuna is, in one screen:

At a glanceTelemeTuna v1.1
InputsLive MQTT off the car's radio link, or CSV files for replays — 15 fields per frame (16 with a timestamp).
OutputsLive Grafana dashboards (gauges, charts, decoded faults, pipeline health), every reading stored in PostgreSQL, plus a full event log.
Runs onSix Docker services started with one command — on a cloud VM (team default, provisioned by Terraform on AWS), a single pit PC, or a hybrid.

1.1 Architecture

💡 The service boxes above — and the names in the table below — are clickable: they jump to that service's developer section.

In words: on the car, the sender ESP32 radios each reading (over LoRa) to the receiver ESP32 in the pit. The receiver hands it to Mosquitto — an MQTT broker running on the computer that works like a post office: the receiver ESP32 publishes messages to it, and it passes them on to whoever subscribes. It’s the bridge between the ESP32 receiver and the rest of the stack. Node-RED is the subscriber — it picks up each message from the broker, stamps it with the arrival time, then cleans, converts, and heals it before writing the result into PostgreSQL. Grafana reads that database and visualizes it in real time (live gauges and charts); pgAdmin lets you open the raw tables by hand. Flyway is what builds those tables in the first place and keeps their version current: each time the program (re)starts it checks the schema, applies any new migration files it hasn’t run yet, brings the tables up to date, then exits — so the database structure stays correct without ever being edited by hand.

2. Getting access

You don't install anything — no Docker, no Git, no terminal. Almost always you just need one thing from whoever runs the platform: the address. Grafana is open to anonymous viewers, so usually there's no login to chase either.

2.1 🌐 If your team hosts on the cloud (Rapidamente's setup)

📊 Grafana — the one thing you actually open

The dashboards live at http://<server-ip>:3001. Anonymous viewing is on — open it and read everything without logging in. A login is only needed to edit dashboards, which is a developer/maintainer task, not a viewer one.

Your team’s server: ask whoever runs the platform for the current Grafana address — a full rebuild can change it.

The rest — viewers barely touch these

2.2 🏠 If your team runs it locally (one PC in the pit)

Two conditions, in order:

  1. 📶 Join the same Wi-Fi / hotspot as the host PC. A device on a different network can't reach it.
  2. 📍 Ask for the host PC's IP on that network — something like 192.168.1.50 — and open Grafana at http://192.168.1.50:3001 (anonymous viewing, no login).
Which setup am I on? An address starting with 192.168. or 10. means local (same-Wi-Fi rule applies). Anything else is the cloud — reachable from anywhere.

If a page won't open

Cloud address dead? Tell your admin which port — usually a security-group/firewall port isn't open or your IP isn't allow-listed (the firewall is scoped to specific IPs).
Local address dead? Work down the list: (1) Are you on the same Wi-Fi / subnet as the host PC? (2) Is the IP still current — a PC's LAN IP can change from one day to the next, so re-check it. (3) Is the host PC awake and running the stack (not asleep or shut down)? (4) Is the host's firewall allowing 3001/1881/5051? — that last one is an admin fix. ⚠️ Windows hosts especially: Windows Defender Firewall blocks incoming connections by default, so teammates often can't see a Windows-hosted dashboard until someone adds an inbound rule for those ports (or allows Docker through the firewall). A Mac host usually works on the LAN without that extra step.

💡 The live dashboard — click the two tabs to switch views, hover any panel to see what it shows, and try the annotation toggles up top (explained in 3.4). Simplified; panel colours are in 3.3.

3.1 Tab 1 · Car Live Dashboard — the car

The car right now: live gauges (RPM, motor & IGBT temperature), stat tiles (voltage, current, torque, drive mode), the six status lights, time-series charts, and an Active Errors & Warnings panel that reads "No active faults — All systems normal" when the car is healthy. Colour meanings: 3.3.

3.2 Tab 2 · TelemeTuna Pipeline Health — the system

The system itself: the Connection fish (is data arriving right now?), the Event Logs feed, counters (healed · warnings · errors · criticals · frame drops · nulls), and Healed Fields (which values were repaired, and when). The fish's three states are explained in 3.3.

THE one rule: every panel except Connection follows the time picker (top-right, default Last 15 minutes). "Live" tiles show the newest reading inside that window; counters count within the window. More on that in 3.5.

The four health levels — what the Event Log is telling you

Every line in the Event Logs (and every counter) is tagged with one of four levels. They describe how the pipeline handled a reading — not a fault the car reported. (The car's own faults are the err/warn bitmasks shown as Active Errors & Warnings on the Car Live tab; those come from the car, these come from TelemeTuna.)

LevelWhen it firesWhat happens to the value
warnA value arrived out of range but is healable (a continuous reading — rpm, temps, volts…)Replaced with the last known-good value (healed); the frame is still stored. A glitch we fixed.
errorA value is wrong and can't be safely guessed — a status light or a fault bitmaskStored as null (blank), never healed — copying an old "no fault" could hide a real one. A value we had to blank out.
FDThe whole frame is unusable — wrong field count, text where a number belongs, or a broken timestampThe entire frame is dropped — nothing from it is stored. A whole reading thrown away.
criticalA software/program bug was caught by the safety net (not bad data — actual code)That frame is skipped, the pipeline keeps running, and the failing station's name is logged. Worth reporting to a developer.
Health levels ≠ car faults. A wall of warn/error/FD means the radio or a sensor is feeding messy data — it does not mean the car flagged a fault. An actual car fault shows up as a red/yellow row in Active Errors & Warnings and as a Car Errors/Warnings annotation. A critical is the odd one out: it's a bug in the code, not the car or the data.

Disconnected — whose fault is it? The fish only knows one thing: did anything reach the platform from the MQTT broker in the last second? So "Disconnected" means the chain feeding the broker is broken somewhere — it's a connection problem, not a dashboard one. A maintainer should debug it backward, from the platform outward: ① is the broker (Mosquitto) healthy and reachable? → ② is the receiver ESP32 on Wi-Fi and publishing to it? → ③ is the sender ESP32 on the car powered and within radio range? The first broken link is the culprit.

3.5 A note on time

Almost every "is it broken?" moment on the dashboard is really about the time window (the picker, top-right). The three that catch people:

A grey "–" tile: there's simply no reading in the selected window — the car stopped, or you're on the wrong range. Honest, not broken.
Counters changed when I zoomed: correct behaviour — zooming is changing the time window they count in.
"No data" when replaying an old session: set the time picker to an absolute range covering that session's date — then the whole dashboard, gauges included, replays it.

The three logins (database · Grafana editing · pgAdmin). Use strong, unique values — they're stored encrypted in AWS SSM Parameter Store and written into the box's .env at boot, never committed. Realistically the only values you must change.

admin_cidr — who can reach the open ports

An IP range (CIDR form) allowed through the firewall to Grafana / Node-RED / pgAdmin / MQTT.
  • Lock it to you: run curl ifconfig.me, then set e.g. 203.0.113.5/32 — only your IP gets in.

Server size and disks. Defaults t3.small / 20 GB / 5 GB are plenty — a 2-hour session is only a few MB. (gp3 disks can grow later, never shrink.)

5.2 ☁️ Cloud by hand — the manual way still supported

Prefer to build the VM yourself — another provider, or AWS without Terraform? Fully viable. Reproduce, by hand, the same settings Terraform applies so the two stay interchangeable:

  1. Create a small VM — Ubuntu 22.04, ~t3.small (1–2 GB RAM) — with an encrypted root disk. Install Docker Engine + the Compose plugin, then do the §4 setup (clone + .env).
  2. Change every password in .env before starting. Bots scan new public IPs within hours.
  3. Give PostgreSQL its own disk (as Terraform does): attach a second encrypted volume, mount it at /mnt/pgdata, bind the database volume to it, and bring up the production stack so data can't be wiped: 
    docker volume create --driver local --opt type=none \
  --opt o=bind --opt device=/mnt/pgdata postgres_data
docker compose -f docker-compose.yaml -f docker-compose.production.yaml up -d
  4. Assign an Elastic / static IP so the address the ESP32 and team use doesn't change on reboot.
  5. Open only these ports, scoped to your IP — done in the cloud firewall's (security group's) inbound rules. This mirrors the Terraform security group exactly:
    PortServiceRestrict to

    1884MQTT — the ESP32 publishes here (host 1884 → 1883 in-container)your IP, or the pit's if static

7.1 Mosquitto (message broker)

MQTT is a publish/subscribe "post office": senders publish to a named topic; subscribers receive everything on topics they follow. Nobody needs anyone's address — only the broker's.

/mosquitto/config/mosquitto.conf
  listener 1883            # in-container port (Compose maps host 1884 → 1883)
  allow_anonymous true     # no username/password (prototype trade-off)
  persistence true         # broker remembers state across restarts
  • Topic: all telemetry rides one topic, car_telemetry.
  • Publisher: the receiver ESP32 (or mosquitto_pub when testing).
  • Subscriber: Node-RED's mqtt in node.
Host 1884, container 1883. The broker listens on 1883 inside Docker; Compose publishes it to the host as 1884:1883. So external publishers (ESP32, mosquitto_pub) use host 1884, while Node-RED — inside the same Docker network — connects to mosquitto:1883 — exactly what the committed flows.json broker node is set to. Changing the port? Keep three things in sync: mosquitto.conf's listener, the Node-RED broker node's port, and both sides of the Compose mapping.

The QoS 2 + persistent-session detail

MQTT has three delivery levels: QoS 0 (fire and forget), QoS 1 (at least once — duplicates possible), QoS 2 (exactly once). Node-RED subscribes at QoS 2 with a persistent session (cleansession: false):

  • If Node-RED is briefly down — typically during a Deploy or container restart — the broker buffers messages and replays them on reconnect, exactly once each.
  • This protects only the broker → Node-RED leg. It does not help the car → broker leg, which has no queue at all (see timestamping).
Does the buffer distort timestamps? In theory a replayed backlog would all get stamped at reconnect. In practice broker and Node-RED share the same Docker network on the same machine, so this leg basically never backlogs outside a deliberate Deploy. The buffer is free insurance. Deploy mid-session and you may see a small same-stamped cluster — the unique-time rule keeps one per millisecond and skips the rest.
If Mosquitto goes down: ESP32 publishes go nowhere (lost); Node-RED's MQTT node auto-reconnects. Honest gap on the charts.
Why the Connection fish counts dropped frames as "connected": it answers "is the link alive?", not "is the data good?" — a malformed frame that arrives still proves the radio + broker path works.
Security: anyone who can reach the published port 1884 can publish. Hardening = broker auth + TLS on 8883.

7.2 Node-RED (pipeline)

Node-RED is a visual programming tool: logic lives in wired-together nodes. Four tabs — Real-Time Imports, CSV Imports, Background Flow (the pipeline), and Test Node (the generator, kept separate so nobody clicks it by accident).

The Background Flow — ten stations, five jobs

The stations group into five jobs, color-coded below and in the diagram: ■ Gatekeeping (1–2: verify structure — nothing inside is trusted yet) · ■ Conversion (3–5: raw numbers → real units) · ■ Repair (6: the only station allowed to fix values) · ■ Strict validation & packing (7–9: discrete signals are never repaired, only accepted or nulled, then packed) · ■ Storage (10: duplicate-proof insert). The order matters: you can't convert what isn't parsed, can't repair what isn't converted, and must never "repair" an alarm.

💡 Click any station box above to open its explanation. (Every station also has a side-exit into the event log.)

1 Strip Timestamp — gatekeeping

16 fields → the first must parse as a valid date (keep it); invalid/blank → drop + FD. 15 fields → stamp arrival time, silently. Anything else → drop + FD. CSV rows strip on the CSV tab (16 required) and join at Station 2 — preserving original times. Why arrival-stamping is correct: timestamping.

2 Parse CSV — gatekeeping

Exactly 15 fields; gear present; everything else a real number — or the whole frame drops + FD naming the field. No repair here: before parsing succeeds, field identity is unknown (a 14-field frame could be missing anything); repair is only safe once identity is certain.

3 Raw → Real — conversion
real = raw ÷ 32767 × max   (5500 RPM · 212.1 A · 200 V · 125 N·m)

Outside ±32767 = didn't come from hardware → null + warn; Station 6 heals it. Rounding is deliberate — six decimals would be fake precision. 🧪 Try it →

4 Map Mode — conversion

N→0 D→1 R→2; anything else → null + warn, healable (a car can't teleport between gears in 300 ms).

5 Temperature Conversion — conversion

IGBT: 32-point calibration table (16308→−30 °C … 28480→125 °C), straight-line interpolation between neighbors. Motor: linear, temp = 30 + (raw−11446)×70÷4554, valid 10000–20000. Out of range → null + warn; never extrapolates a guess. 🧪 Try it →

6 Heal — repair

Keeps a last-known-good snapshot of 7 fields; plausibility windows: rpm ±6000 · amp ±250 · volt ±250 · trq ±150 · temps −40…200 °C · mode 0–2. Valid → keep & update snapshot. Invalid/null → replace from snapshot, record the name. Full reasoning: healing logic.

7 Flags — strict validation

Exactly 0/1 → false/true; anything else → null + error. Never healed — see healing logic.

8 Validate Bitmasks — strict validation

Whole numbers 0–65535 or null + error. Stored raw; Grafana decodes (how). 🧪 Build a bitmask →

9 Build Parameters — packing

Packs timestamp + 4 motion + gear + 2 masks + 2 temps + 6 booleans + healed-fields list, in exact column order.

10 Database Write — storage
INSERT INTO telemetry_records (…) VALUES (…) ON CONFLICT (time) DO NOTHING;

Unique time → replays can never double-count (PostgreSQL).

The crash net — and its one deliberate blind spot

Expected problems ride each station's second output into the log. Unexpected bugs are caught by a global Catch node per tab → critical with the station's name — and the pipeline keeps running for the next frame.

The blind spot, by design: the Catch nodes do not watch "Normalize Log Event" / "Log Database". If the DB were down, a caught log-write failure would trigger another log write → fail → forever. Excluding the logging chain breaks the loop; the cost is that errors during a DB outage go unrecorded. Docker restarts the DB automatically.
If the database goes down: telemetry inserts fail and those readings are lost — there's no buffer between Node-RED and PostgreSQL.

7.2.1 Input flows & data input — every road in, and how to feed it

Four ways in. Note the two different entry points: live sources enter at Station 1 (they need timestamping), while raw CSVs strip their timestamp on the CSV tab — with a stricter version of Strip Timestamp that requires 16 fields — and enter at Station 2. Processed CSVs skip the pipeline entirely.

  • Live sources (MQTT, serial, test generator) send 15-field frames → link in 1 → Station 1 stamps them on arrival.
  • Raw CSV uses its own, stricter Strip Timestamp (16 fields or the row is dropped — historical data must carry its own time) → link in 2 → enters at Station 2, keeping the file's original timestamps.
  • Processed CSV never touches the Background Flow: parse, then straight to the database. Already-cleaned data must not be converted or healed twice.

Feeding data in — the three ways

The flows above are the roads in; here is how you actually push data down them. Live MQTT is the racing default, CSV replays an old session, and serial is the local USB option.

① Live MQTT — the racing path. Node-RED always listens on topic car_telemetry. The receiver ESP32 publishes 15-field frames to the broker (cloud IP or local host, host port 1884); they flow in and get stamped on arrival — nothing to click. Hand-test from any machine:

mosquitto_pub -h <server-ip> -p 1884 -t car_telemetry \
  -m "15000,-8000,19660,12000,D,21357,11644,0,0,0,0,0,1,0,1"
1884 or 1883? External publishers (the ESP32, mosquitto_pub) connect to the host port 1884. Inside Docker the broker listens on 1883, so Node-RED's own MQTT node uses mosquitto:1883 — Compose maps 1884:1883 between them. More in 7.1 Mosquitto.

② CSV import — replays. Two ideas: drop the file in, then point Node-RED at it. Both CSV types need a timestamp as the first column (16 columns total):

  • Raw CSV — timestamp + 15 raw fields → full cleaning pipeline (original times preserved).

7.2.2 Timestamping — who stamps, and why

A real engineering decision, documented inside the flow as a "Note to Future Developer": the ESP32 sends no timestamps; Node-RED stamps each frame on arrival, silently.

Why that's accurate: MQTT guarantees delivery only between the broker and the platform. It does not buffer on the ESP32 — if the car's link drops, readings made while offline are lost, never queued, never flushed later in a burst. Everything that arrives was sent in real time, so arrival timestamps don't condense or distort the charts.

The honest cost: radio dropouts = lost data and an honest gap — not delayed data. The future path: to survive disconnects, two changes must ship together — ① firmware buffering on the ESP32 and ② sender timestamps attached at measurement time (RTC or NTP). A flushed backlog arrives in seconds; only sender timestamps preserve when each reading actually happened. The pipeline already accepts the 16-field format, so that upgrade needs zero Node-RED changes.

7.3 Flyway (migrations)

Flyway answers one question: how does the database get its tables, reliably, on any machine? The answer is migrations — versioned SQL files applied once each, in order.

  • On startup Flyway checks its history table (flyway_schema_history — stored in the database itself) and runs only what hasn't run before, in version order.
  • Then it exits. "Flyway exited" in docker compose ps is success — verify with docker compose logs flyway.
  • The golden rule: never edit an applied migration — add a new, higher-numbered V file. Any fresh machine reaches the identical schema by replaying the same files.

8. Infrastructure as code (Terraform & AWS)

Maintainers only need terraform apply (§5.1). This section is the other half: how the code actually works, module by module. Everything lives in the repo's infrastructure/ folder.

Why infrastructure-as-code? Instead of clicking around the AWS console, the entire server is described in files — reproducible, reviewable in git, and destroyable in one command. Terraform reads the desired state, diffs it against what already exists, and makes only the changes needed.

8.1 The four modules

A small root module wires four child modules together. Terraform infers the dependency order automatically from the references between them (secrets → iam, network → compute, iam → compute) — no manual ordering.

infrastructure/
├── versions.tf      # Terraform ≥ 1.5, AWS provider ~> 5.0
├── providers.tf     # AWS provider + default_tags (Project/Environment/ManagedBy)
├── variables.tf     # every input (region, sizes, credentials…)
├── main.tf          # wires the four modules below together
├── outputs.tf       # instance_id, public_ip, grafana_url
├── terraform.tfvars.example
├── .terraform.lock.hcl     # provider version lock (aws 5.100.0)
└── modules/
    ├── network/   # default VPC/subnet lookup + the security group
    ├── secrets/   # app credentials → SSM Parameter Store (7 params)
    ├── iam/       # EC2 role: SSM access + read-only on its own secrets
    └── compute/   # the EC2 instance, data volume, Elastic IP, boot script

🌐 network — the firewall

Looks up your account's default VPC and its first default subnet (and that subnet's availability zone, because the data volume must live in the same AZ). Creates the security group telemetuna-prod-sg with one ingress rule per published port, each scoped to admin_cidr:

Egress is fully open (so the box can pull Docker images, OS updates, and reach SSM). There is deliberately no port 22 / SSH rule.

🔐 secrets — credentials in SSM Parameter Store

Writes seven parameters under the path /<project>/<environment>/… (e.g. /telemetuna/prod/postgres_password). Usernames, the DB name, and the pgAdmin email are plain String; the three passwords are encrypted SecureString. The module outputs the parameter ARNs so the IAM policy can be scoped to exactly these and nothing else.

TypeParameters
String ×4postgres_user · postgres_db · grafana_admin_user · pgadmin_email

👤 iam — least-privilege identity

Creates the role the EC2 instance assumes. It attaches the AWS-managed AmazonSSMManagedInstanceCore policy (enables Session Manager) and an inline policy allowing ssm:GetParameter/GetParameters on only the seven parameter ARNs above, plus kms:Decrypt restricted (via a kms:ViaService condition) to SSM in this region. The role is exposed through an instance profile.

🖥️ compute — the server itself

  • Looks up the latest Ubuntu 22.04 (Jammy) AMI from Canonical — no hard-coded AMI IDs.
  • Creates a dedicated encrypted gp3 EBS volume for Postgres data (tagged Backup = postgres), in the subnet's AZ, attached at /dev/sdf.
  • Launches the EC2 instance with a public IP, an encrypted gp3 root disk, and IMDSv2 required (http_tokens = "required").
  • Runs the boot script (user_data.sh.tftpl) on first launch — see 9.2.
  • Creates an Elastic IP and associates it. ⚠️ terraform destroy releases this address, so a full rebuild changes the public IP — use stop/start day to day.

8.2 The first-boot script (user_data.sh.tftpl)

The instance configures itself on first boot in nine logged steps — you never SSH in to set it up (and there's no SSH to use):

  1. Install Docker + Compose plugin.
  2. Install AWS CLI v2 (so it can read SSM).
  3. Mount the dedicated Postgres data volume — find the non-root disk, format ext4 if blank, add to /etc/fstab (mounted at /mnt/pgdata), and remove lost+found so Postgres can initdb into a clean folder.
  4. Read config + secrets from SSM Parameter Store (decrypting the passwords).
  5. Clone the repo into /opt/Rapidamente-TelemeTuna.
  6. Hand the Node-RED folder to uid 1000 so the container can write to its /data mount.
  7. Bind the postgres_data Docker volume to the EBS disk (/mnt/pgdata).
  8. Write .env entirely from SSM (umask 077) — production does not depend on .env.example.
  9. Write helper scripts & auto-start the stack — creates start-telemetuna.sh and update-telemetuna.sh, then brings the stack up with both compose files.
Self-starting by design. Every service uses restart: unless-stopped, so the stack comes back by itself after a reboot or stop/start without re-running the script. Two helper scripts stay on the box: sudo /opt/Rapidamente-TelemeTuna/start-telemetuna.sh (bring it up) and sudo /opt/Rapidamente-TelemeTuna/update-telemetuna.sh (git pull + rebuild — to ship app updates).

12.4 Editing dashboards — local dev first

Never live-edit production Grafana. Grafana re-provisions its dashboards from the repo on every start, so a change made directly in production is lost on the next rebuild. The safe workflow keeps it versioned in git:
  1. Bring up your local dev stack (docker compose up -d) and open Grafana at localhost:3001.
  2. Log in with the Grafana admin creds from your .env and edit the dashboard there.
  3. Export the dashboard JSON (Dashboard settings → JSON Model) and save it over services/grafana/provisioning/dashboards/car-telemetry.json.
  4. Commit and ship with update-telemetuna.sh — production re-provisions from the file.

Same idea for the Node-RED pipeline: edit on dev, the flow saves into services/nodered/flows.json, commit, ship. CSV imports are a trusted-party-only task — see §5.2.
